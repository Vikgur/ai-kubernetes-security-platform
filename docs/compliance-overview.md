# Назначение Compliance Overview

`compliance-overview.md` описывает **подход к compliance как архитектурному свойству** Kubernetes-платформы Sovereign AI.

Документ отвечает на вопросы:
- почему compliance встроен в платформу;
- какие стандарты покрываются;
- как осуществляется трассировка контролей;
- как генерируются доказательства соответствия.

Compliance здесь — не отчёт.  
Это **побочный эффект корректной архитектуры**.

---

# Базовый принцип

Compliance не добавляется после.  
Он **возникает автоматически**, если платформа спроектирована правильно.

Подход:
- сначала архитектура;
- затем security-инварианты;
- затем policies и audit;
- и только потом отчёты.

Документация без enforcement не считается compliance.

---

# Почему compliance встроен

Причины:
- sovereign и regulated среды не допускают post-hoc проверок;
- требования должны быть enforce’имы, а не описаны;
- человеческий фактор исключается.

Compliance без автоматизации:
- не масштабируется;
- не воспроизводится;
- не выдерживает аудит.

---

# Поддерживаемые стандарты

Платформа ориентирована на:

- **CIS Kubernetes Benchmark**  
- **NIST (800-53 / 800-190)**  
- **ISO/IEC 27001**

Стандарты не реализуются напрямую.  
Они **маппятся на архитектурные контроли**.

---

# CIS Coverage

CIS используется как:
- технический baseline;
- источник контрольных требований;
- ориентир для hardening.

Контроли CIS:
- реализуются через policies;
- проверяются автоматически;
- не требуют ручной валидации.

---

# NIST Coverage

NIST используется как:
- модель управления рисками;
- framework для governance;
- основа для sovereign требований.

Контроли NIST:
- трассируются к security layers;
- выражаются через policies и audit;
- используются для risk-based анализа.

---

# ISO 27001 Coverage

ISO рассматривается как:
- управленческий и процессный стандарт;
- надстройка над техническими мерами.

Платформа:
- поставляет технические доказательства;
- снижает объём ручных процедур;
- упрощает сертификацию.

---

# Mapping Approach

Подход к маппингу:

- стандарт → контроль  
- контроль → архитектурный элемент  
- элемент → policy / audit / runtime  

Каждый контроль:
- имеет источник;
- имеет реализацию;
- имеет проверяемое доказательство.

Mapping хранится в коде, а не в таблицах.

---

# Evidence Generation

Evidence генерируется автоматически:
- audit logs;
- policy evaluation results;
- CI compliance checks;
- runtime alerts.

Evidence:
- воспроизводимо;
- привязано к версии Git;
- готово к внешнему аудиту.

---

# Compliance в CI

CI:
- валидирует соответствие политикам;
- проверяет покрытие контролей;
- формирует отчёты.

Если CI зелёный — кластер соответствует заявленному уровню.

---

# Связь с compliance/*

Каталог `compliance/` содержит:
- mapping стандартов;
- реализованные контроли;
- шаблоны отчётов.

Каждый файл:
- связан с архитектурным решением;
- имеет enforce-механизм.

---

# Связь со структурой репозитория

Compliance не вынесен отдельно.  
Он пронизывает:

- policies;
- runtime security;
- audit;
- CI.

Подробно: `docs/repository-structure.md`

---

# Инварианты compliance-подхода

Никогда не нарушаются:
- нет ручного подтверждения;
- нет деклараций без enforcement;
- нет среды без audit;
- нет compliance без Git.

---

# Не-цели compliance

Платформа не предназначена для:
- ручного заполнения чеклистов;
- one-off сертификаций;
- “бумажного” соответствия;
- разовых аудитов.

---

# Итог

Compliance в Sovereign AI:
- следует из архитектуры;
- проверяется автоматически;
- масштабируется вместе с платформой;
- выдерживает external audit.

Если систему можно сломать без следа — она не compliant.
